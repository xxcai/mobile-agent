# Mobile Agent 协作流程

在本仓库中协作开发时，遵循以下方式。

## 核心规则

- 开始修改前，先阅读当前相关代码路径。
- 先写或更新 `.planning/feature/` 下的规划文件。
- 规划采用“总-分”结构：
  - 一个总览文件或目录级 `README.md`
  - 每个实现步骤一个独立文件
- 每一步都必须足够小、可测试、可单独评审。
- 未经用户明确同意，不进入下一步。
- 未经用户明确同意，不提交代码。

## 规划规则

- 优先使用 `.planning/feature/<feature-name>/` 目录。
- 总览文件应说明：
  - 目标
  - 关键决策
  - 步骤顺序
- 每个步骤文件应说明：
  - 目标
  - 范围
  - 不修改的内容
  - 测试方法
  - 验收标准
- 如果当前 feature 未收口，但用户插入了一个独立任务，应为该任务新建独立的 `.planning/feature/<new-feature>/` 目录，不要混入原 feature 的步骤目录。
- 插入任务完成后，如果用户仍要继续原 feature，再回到原规划继续推进。
- 需求变化时，先更新规划，再改实现。
- `.planning` 仅供本地协作参考，不提交到 git。

## 实施规则

- 每次只实现当前已经获批的那一步。
- 不要把多个性质不同的改动混在同一步里。
- 除非用户明确同意，否则优先保持兼容。
- 如果要引入新架构，先保留兼容路径，再逐步迁移。
- 遇到设计分歧时，先说明取舍，再动代码。
- 实施过程中要控制提交边界，避免把无关改动一起带入。

## 测试规则

- 优先选择能证明当前步骤的最小测试面。
- 如果需要人工验证，优先把测试入口放在 `app/` 层，减少对库模块的测试侵入。
- 测试页面应清晰展示：
  - 测试项
  - 输入
  - 实际结果
  - 预期结果
  - PASS / FAIL

## 提交规则

- 每一步完成后：
  - 先汇报改了什么
  - 再汇报如何验证
  - 再说明当前提交边界
- 提交代码时，默认不要把 `.planning/` 文件加入 commit，除非用户明确要求把规划也纳入版本控制。
- 汇报提交边界时，应明确说明本次提交是否包含 `.planning`、文档或仅包含代码。
- 等用户确认后，才允许提交。
- 等用户确认后，才允许进入下一步。
- 如果文档需要同步，应在功能阶段收口时统一处理。

## 文档规则

- `docs/` 用于对外或对项目协作者可见的正式文档。
- `.planning` 用于本地协作和过程记录。
- 文档内容应与当前代码状态一致，避免保留过时说明。
- 如果改动影响外部接入方式、对外 API、工具契约或正式使用方式，应在规划阶段把文档更新单独列成一个实现步骤，而不是只在最后口头补充。
- 功能闭环后，应检查是否需要同步 `docs/`。

## 沟通方式

- 解释要直接、技术化、可执行。
- 先说明边界和假设，再说明方案。
- 不把临时需求细节固化为长期协作规则。
- 如果需求变化，先更新规划文件，再开始实现。
