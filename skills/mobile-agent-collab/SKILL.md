# Mobile Agent 协作流程

在本仓库中进行功能开发时，遵循以下流程。

## 核心规则

- 开始修改前，先阅读当前相关代码路径。
- 先写或更新 `.planning/feature/` 下的规划文件。
- 规划采用“总-分”结构：
  - 一个总览文件或目录级 `README.md`
  - 每个实现步骤一个独立文件
- 每一步都必须足够小、可测试、可单独评审。
- 未经用户明确同意，不进入下一步。
- 未经用户明确同意，不提交代码。

## 规划规则

- 优先使用 `.planning/feature/<feature-name>/` 目录。
- 总览文件应说明：
  - 目标
  - 协议或架构决策
  - 步骤顺序
- 每个步骤文件应说明：
  - 目标
  - 精确范围
  - 不修改的内容
  - 测试方法
  - 验收标准

## 实施规则

- 每次只实现当前已经获批的那一步。
- 不要把框架改造、运行时实现、提示词优化混在同一步里。
- 除非用户明确同意，否则优先保持兼容。
- 如果要引入新架构，先保留兼容路径，再逐步迁移。

## 测试规则

- 优先选择能证明当前步骤的最小测试面。
- 如果用户要求测试不要修改库模块，就把测试入口放在 `app/`。
- 如果用户要求，可以长期使用 `MainActivity` 作为手工测试入口。
- 测试页面应清晰展示：
  - 当前测试用例
  - 输入请求
  - 实际结果
  - 预期结果
  - PASS / FAIL

## 提交规则

- 每一步完成后：
  - 先汇报改了什么
  - 再汇报验证了什么
  - 然后等待用户确认
- 只有在用户明确同意后，才可以：
  - 提交当前步骤
  - 进入下一步

## 沟通风格

- 解释要直接、技术化、少空话。
- 遇到设计分歧时，先说明取舍，再动代码。
- 如果需求变化，先更新规划文件，再开始实现。

## 本仓库当前偏好

- `call_android_tool` 是兼容通道，在重构过程中应优先保留。
- 新能力域适合拆成独立的外层通道，而不是继续堆进旧通道。
- 如果用户要求减少对 `agent-core` / `agent-android` 的测试侵入，手工验证入口可以放在 `MainActivity`。
