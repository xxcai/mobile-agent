# 打卡助手

帮助 Agent 在宿主 App 中完成考勤打卡。标准路径是：

1. 回到首页
2. 点击底部 Tab 中的“业务”
3. 在业务页面找到“打卡”卡片
4. 进入打卡页面后点击“打卡”按钮
5. 观察打卡结果并向用户反馈

## 触发条件

此技能应该在以下情况下触发：

- 用户说“打卡”
- 用户说“帮我打卡”
- 用户说“上班打卡”
- 用户说“下班打卡”
- 用户说“考勤打卡”
- 用户说“签到”
- 用户要求进入打卡、考勤或签到入口并完成操作

不要在用户只是想查看业务页、查看考勤规则、查询历史打卡记录时直接点击打卡按钮；这类任务需要先确认用户是否真的要执行打卡动作。

## 核心路径

固定导航路径如下：

1. 首页 `MainActivity`
2. 底部 Tab “业务”
3. 业务页面中的“打卡”卡片
4. 打卡页面中的“打卡”按钮
5. 打卡结果提示

如果当前页面不是首页，不要猜测当前页面上的业务入口。应先通过页面已有的返回、关闭、取消等可见安全入口回到首页，再执行固定路径。

## 工作流程

### 步骤 1：先看当前页面

优先调用 `android_view_context_tool`，判断当前页属于哪一类：

- 首页
- 业务页
- 打卡页
- 打卡结果页
- 其他页面

建议调用：

```json
{
  "targetHint": "首页、底部业务Tab、业务页面打卡卡片或打卡按钮"
}
```

如果当前已经在打卡页：

- 不要重复进入业务页
- 直接寻找打卡按钮

如果当前已经显示“打卡成功”“已打卡”或明确结果：

- 不要重复点击打卡
- 直接把结果反馈给用户

### 步骤 2：如果不在首页，先退回首页

如果当前页面既不是首页，也不是业务页、打卡页或打卡结果页：

1. 优先使用页面上的返回、关闭、取消等安全入口回退
2. 回退动作必须通过 `android_view_context_tool` 找到可见入口，再用 `android_gesture_tool` 的 `tap` 执行
3. 每次点击返回或关闭入口后，必须重新调用 `android_view_context_tool`
4. 最多连续回退 3 次
5. 如果页面上没有明确返回/关闭入口，或 3 次后仍无法确认到达首页，停止并向用户说明当前页面无法安全回到首页

不要盲目点击页面中部卡片、广告、列表项或未知入口。

### 步骤 3：从首页进入业务页

如果当前页面是首页，并且底部 Tab 中可见“业务”：

1. 从最新 observation 中找到“业务”Tab
2. 使用该元素的 bounds 计算中心点
3. 调用 `android_gesture_tool` 执行 `tap`
4. `tap` 参数必须包含：
   - `x`
   - `y`
   - `observation.snapshotId`
   - `observation.referencedBounds`
   - `observation.targetDescriptor`

点击后必须重新调用 `android_view_context_tool`，确认已经进入业务页面。

如果底部 Tab 中没有看到“业务”，不要猜测坐标。

### 步骤 4：从业务页进入打卡页

如果当前页面是业务页：

1. 找到“打卡”“考勤打卡”“签到”等入口卡片
2. 优先选择可点击父容器或融合后的 `fused/native` 候选
3. 使用候选 bounds 计算点击点
4. 调用 `android_gesture_tool` 执行 `tap`
5. 点击后重新观察，确认进入打卡页面

如果页面中有多个相似入口：

- 优先选择标题或主文案精确等于“打卡”的卡片
- 不要选择历史记录、规则说明、统计报表等非执行入口
- 如果仍不唯一，向用户确认或回退到模型决策，不要本地盲点

如果当前业务页首屏没有看到“打卡”卡片或打卡按钮，可以受控滚动页面继续寻找。这里要查看更多下方业务卡片，调用 `android_gesture_tool` 时使用 `direction=down`，表示“内容向下滚动/露出更下方内容”；runtime 实际会注入手指上划动作。不要使用 `direction=up`，因为它会向页面顶部滚动，在顶部附近可能触发下拉刷新。

1. 先从最新 observation 中明确选择业务页面的主要滚动容器
2. 调用一次 `android_gesture_tool`
   - `action=swipe`
   - `direction=down`
   - `amount=small` 或 `medium`
   - `observation.snapshotId`
   - `observation.referencedBounds`
3. 滑动后必须重新调用 `android_view_context_tool`
4. 如果新页面中出现“打卡”“考勤打卡”“签到”等入口，再按上面的点击规则进入

滑动限制：

- 最多向下滚动 2 次（即实际手势上划 2 次）
- 每次滑动后都必须重新观察页面
- 如果滑动后页面没有新增内容或仍没有打卡入口，应停止继续滑动
- 不要下拉刷新
- 不要来回反向滑动
- 不要在未明确滚动容器时盲滑

### 步骤 5：点击打卡按钮

如果当前页面是打卡页，并且可见明确的“打卡”按钮：

1. 确认按钮不是“查看记录”“打卡规则”“重新定位”等非打卡动作
2. 找到“打卡”“立即打卡”“上班打卡”“下班打卡”等按钮
3. 调用 `android_gesture_tool` 执行 `tap`
4. 点击后重新调用 `android_view_context_tool`
5. 根据结果提示向用户反馈

点击打卡按钮后，如果重新观察到“打卡时间”已经更新，或页面显示新的打卡时间、已打卡、打卡成功等状态，就表示打卡已经成功完成。此时不要再次点击打卡按钮，也不要为了重复确认而多次点击。

如果页面显示需要定位、权限、网络、打卡范围等前置条件：

- 不要绕过权限或强行继续点击
- 根据页面提示向用户说明失败原因或需要用户确认的动作

## 成功与失败判断

### 成功信号

看到以下任一信息时，可以认为打卡完成或已处于完成状态：

- 打卡成功
- 已打卡
- 已完成打卡
- 成功
- 打卡时间
- 打卡时间已更新
- 页面显示新的打卡时间
- 上班打卡成功
- 下班打卡成功

### 失败或阻断信号

看到以下信息时，应停止盲目重试，并向用户说明：

- 打卡失败
- 不在打卡范围
- 定位失败
- 网络异常
- 无权限
- 未到打卡时间
- 需要开启定位
- 需要登录或重新认证

## 安全规则

- 点击前必须先观察页面
- 动作必须基于最新 `snapshotId`
- 不要使用裸坐标猜测
- 不要在不确定的页面里直接点击“打卡”
- 不要连续重复点击同一个打卡按钮
- 只要打卡时间已更新或出现已打卡状态，就停止继续点击
- 不要为了打卡绕过权限、定位、认证或公司策略提示
- 页面不推进时最多重试一次观察，不要无限循环

## 输出要求

完成后向用户反馈当前结果，尽量简洁：

```text
已完成打卡，页面显示：打卡成功。
```

如果失败或被阻断：

```text
当前没有完成打卡，页面提示：定位失败。需要先确认定位权限或网络状态。
```

如果无法安全到达首页或找不到入口：

```text
我没有找到稳定的打卡入口，已停止继续点击，避免误操作。
```
